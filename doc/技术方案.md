# Static Server Lite 技术方案

## 1. 项目概述

一个轻量级静态资源服务，提供两类能力：

- **公开访问**：匿名用户通过 URL 访问指定文件夹下的静态资源（图片、文档等），但不暴露目录列表。
- **管理后台**：管理员登录后，可在 Web 界面对资源进行文件夹/文件的增删改查（上传、删除、新建文件夹等）。

### 1.1 核心约束

| 维度 | 约束 |
| --- | --- |
| 静态资源访问 | 无需鉴权；禁止目录列表 |
| 管理操作 | 必须登录（JWT） |
| 上传 | 支持多文件并发；上传期间 UI 锁定 |
| 文件夹删除 | 仅允许删除空文件夹 |
| 路径安全 | 所有路径必须校验，禁止跨越根目录（防 `../` 穿越） |

## 2. 技术选型

| 层 | 选型 | 理由 |
| --- | --- | --- |
| 服务端框架 | Koa 2 | 轻量、中间件机制清晰 |
| 鉴权 | JWT (jsonwebtoken) | 无状态、易扩展、前后端分离友好 |
| 密码哈希 | bcrypt | 单管理员账号在配置文件中以 bcrypt hash 存储 |
| 文件上传 | @koa/multer | 流式上传、支持多文件、字段级控制 |
| 路由 | @koa/router | Koa 官方路由 |
| 静态托管 | koa-send（按需） | 显式指定文件路径下发，避免 koa-static 默认的目录索引 |
| 配置管理 | dotenv + 默认配置合并 | 通过 `.env` 与 `config/default.js` 维护 |
| 日志 | pino | 结构化、性能好 |
| 前端框架 | React 18 + Vite | 开发体验好、构建快 |
| UI | Ant Design 5 | 表格/上传/弹窗组件齐全，与管理后台契合 |
| 状态管理 | Zustand（轻量） | 比 Redux 简洁，适合小型管理台 |
| 路由 | React Router 6 | 标准选型 |
| HTTP 客户端 | axios | 拦截器统一注入 token、处理 401 |
| 测试（后端） | Vitest + Supertest | 与前端共享测试栈，supertest 跑 HTTP 集成测试 |
| 测试（前端） | Vitest + React Testing Library + MSW | 组件测试 + 接口 mock |
| 测试覆盖率 | v8 (Vitest 内置) | 输出到 `coverage/` |
| 包管理 | pnpm（推荐）或 npm | monorepo 风格管理前后端 |

## 4. 配置设计

### 4.1 配置项

| Key | 默认值 | 说明 |
| --- | --- | --- |
| `PORT` | 3000 | 服务端口 |
| `STATIC_ROOT` | `./public` | 静态资源根目录（用户指定的"静态资源文件夹"） |
| `JWT_SECRET` | （必填） | JWT 签名密钥 |
| `JWT_EXPIRES_IN` | `2h` | Token 过期时间 |
| `ADMIN_USERNAME` | `admin` | 管理员用户名 |
| `ADMIN_PASSWORD_HASH` | （必填） | bcrypt 哈希后的密码 |
| `MAX_UPLOAD_SIZE` | 50MB | 单文件大小上限 |
| `LOG_LEVEL` | `info` | 日志级别 |

提供 `pnpm gen:hash <password>` 脚本，便于初始化 `ADMIN_PASSWORD_HASH`。

### 4.2 安全约束

- `STATIC_ROOT` 启动时必须解析为绝对路径并校验存在；不存在则启动失败。
- `JWT_SECRET` 启动时校验长度 ≥ 32 字符，否则告警/拒绝启动（生产环境）。

## 5. 接口设计

所有接口统一返回：

```json
{ "code": 0, "data": {...}, "message": "ok" }
```

错误响应：`code !== 0`，HTTP 状态码同步语义化（401/403/404/400/500）。

### 5.1 公开接口

静态资源**直接挂载在根路径**，访问 URL 不带前缀，例如：

- `STATIC_ROOT/foo.png`     → `GET /foo.png`
- `STATIC_ROOT/imgs/a.jpg`  → `GET /imgs/a.jpg`

| Method | Path | 说明 |
| --- | --- | --- |
| GET | `/*`（兜底） | 在 STATIC_ROOT 下查找对应文件并下发；目录/不存在/被保留前缀拦截 → 404 |

实现要点见 §6.7（根路径托管与保留前缀）。

### 5.2 鉴权接口

| Method | Path | 鉴权 | 说明 |
| --- | --- | --- | --- |
| POST | `/api/auth/login` | 否 | body: `{username, password}`；返回 `{token, expiresAt}` |
| POST | `/api/auth/verify` | 是 | 验证当前 token 是否有效（前端启动时调用） |

### 5.3 管理接口（均需 `Authorization: Bearer <token>`）

| Method | Path | 说明 |
| --- | --- | --- |
| GET | `/api/admin/list?path=<相对路径>` | 列出指定路径下的文件与文件夹（含大小、mtime、type） |
| POST | `/api/admin/folder` | body: `{path, name}`；新建文件夹 |
| DELETE | `/api/admin/folder` | body: `{path}`；删除空文件夹（非空返回 400） |
| DELETE | `/api/admin/file` | body: `{path}`；删除单个文件 |
| POST | `/api/admin/files/batch-delete` | body: `{paths: string[]}`；批量删除文件 |
| POST | `/api/admin/upload` | multipart/form-data；字段：`path`（目标目录）+ `files`（多文件）；返回每个文件的成功/失败结果 |

所有 `path` 入参均需经过 `safePath` 校验，越界返回 400。

## 6. 关键实现要点

### 6.1 路径穿越防御 (`safePath`)

```js
// 伪代码
function resolveSafe(staticRoot, relPath) {
  const abs = path.resolve(staticRoot, relPath || '.');
  const rootAbs = path.resolve(staticRoot);
  if (abs !== rootAbs && !abs.startsWith(rootAbs + path.sep)) {
    throw new BadRequestError('PATH_OUT_OF_ROOT');
  }
  return abs;
}
```

- 所有涉及路径的接口必须先调用此函数。
- 同时拒绝包含 `\0`、绝对路径、Windows 盘符等异常输入。

### 6.2 禁止目录列表

- 公开 `/files/*` 路径只用 `koa-send`，从不进入 `serve-index` 之类的中间件。
- 若解析结果是目录 → 直接 404，不暴露"是目录"的信息。

### 6.3 上传流程

1. 前端：`UploadArea` 接收多文件，点击"上传"后立即进入 `uploading` 状态：
   - 给区域加 `pointer-events: none` + 半透明遮罩 + Spin。
   - 禁用其他操作按钮（删除、新建文件夹、面包屑跳转）。
2. 使用单个 `multipart/form-data` 请求带多文件（axios 的 `FormData`，监听 `onUploadProgress` 显示总进度）。
3. 后端：`@koa/multer` 配置 `dest` 临时目录 + `limits.fileSize`，上传完成后 `fs.rename` 到目标目录；目标已存在则按策略（默认追加 `_1`、`_2` 后缀）。
4. 响应包含每个文件的 `{name, success, error?}` 列表，前端按结果提示。

### 6.4 鉴权中间件

- 解析 `Authorization: Bearer xxx`，`jsonwebtoken.verify` 失败统一 401。
- 登录接口对密码做 bcrypt 比对；为防爆破，加 `LRU` 限流：同 IP 5 次失败/分钟则 429。

### 6.5 错误处理

统一中间件捕获：
- 自定义错误类：`BadRequestError(400)`、`UnauthorizedError(401)`、`NotFoundError(404)`。
- 未捕获异常 → 500，pino 输出 stack；响应不暴露内部细节。

### 6.6 跨平台路径

后端统一使用 `path.posix` 处理 API 入参（前端传过来的都是 `/` 分隔），落盘时用 `path.resolve` 自动适配 OS。

### 6.7 根路径托管静态资源 + 保留前缀

为了让用户直接通过 `https://host/foo.png` 访问 `STATIC_ROOT/foo.png`（而不是 `/files/foo.png`），后端按以下**路由优先级**装配中间件，先匹配先生效：

| 优先级 | 路径 | 处理 |
| --- | --- | --- |
| 1 | `/api/*`         | 业务 API（鉴权、管理、登录） |
| 2 | `/admin`、`/admin/*` | 管理后台 SPA（`web/dist`，history fallback 到 `index.html`） |
| 3 | 其他所有 GET/HEAD | 兜底：在 `STATIC_ROOT` 下查找文件，命中则下发，否则 404 |

> **根路径 `/` 不做重定向**：访问 `/` 会落到优先级 3 的兜底中间件，去 `STATIC_ROOT` 找根级文件（如 `index.html`），未命中返回 404。如需访问管理后台请直接访问 `/admin`。

**保留前缀（Reserved Prefixes）**

`/api`、`/admin` 是保留前缀，由路由优先级保证它们永远走前面的中间件。

**不做主动校验**：如果 `STATIC_ROOT` 下存在同名顶级目录或文件（例如 `STATIC_ROOT/api/x.png`），它们会被前面的路由遮蔽而无法通过 URL 访问 —— 这是已知行为，由用户自行规避，后端不做启动扫描、不在新建/上传接口里拒绝。

理由：
- 降低实现与文档复杂度；
- 这种命名冲突极少见，且后果是"访问不到"而非"安全问题"；
- 管理后台仍可正常列出、删除这些条目（管理接口走的是 `/api/admin/*`，操作的是 FS，不受 URL 路由影响）。

保留前缀列表后续可通过配置项 `RESERVED_PREFIXES` 扩展（如新增 `/health`、`/metrics`），扩展时同样不做存量校验。

**兜底中间件实现要点（伪代码）**

```js
const RESERVED = ['/api', '/admin'];

app.use(async (ctx, next) => {
  if (ctx.method !== 'GET' && ctx.method !== 'HEAD') return next();
  // 让保留前缀走前面的路由
  if (RESERVED.some(p => ctx.path === p || ctx.path.startsWith(p + '/'))) {
    return next();
  }
  // 路径安全校验
  const rel = decodeURIComponent(ctx.path);
  let abs;
  try { abs = resolveSafe(STATIC_ROOT, rel); }
  catch { ctx.status = 400; return; }

  const stat = await fs.promises.stat(abs).catch(() => null);
  if (!stat || !stat.isFile()) { ctx.status = 404; return; } // 目录/不存在统一 404，不暴露目录信息

  await send(ctx, rel, {
    root: STATIC_ROOT,
    hidden: false,   // 拒绝 dotfile（.env / .git 等）
    index: false,    // 不自动找 index.html
  });
});
```

**安全注意点（根路径托管比前缀方案风险更高）**

- `safePath` 校验仍然必须，防 `..` 穿越。
- `hidden: false` 兜底拒绝 dotfile，避免管理员误把项目根设为 `STATIC_ROOT` 时泄漏 `.env`、`.git/config`。
- 解析为目录一律 404（不返回 301 加斜杠，避免暴露"是目录"的信息）。
- 仅响应 `GET`/`HEAD`，其他方法直接交给后续路由（避免 `PUT /foo.png` 这类被误以为是写操作）。

## 7. 前端设计

### 7.1 路由

| Path | 页面 |
| --- | --- |
| `/admin/login` | 登录页 |
| `/admin` | 管理主页面（受保护，未登录跳登录） |

### 7.2 主要组件

- **Breadcrumb**：当前路径分级展示，点击任意级跳转；提供"上一级"按钮。
- **FileTable**：列表展示文件/文件夹；勾选支持批量；
  - 文件夹行：双击或点击进入；操作列只有"删除"（空才能删）；"访问"列显示 `-`（静态服务无目录索引）。
  - 文件行：操作列有"删除"；"访问"列提供"Open"链接，`target="_blank"` + `rel="noopener noreferrer"`，新标签打开静态 URL（路径按 `/` 分段 `encodeURIComponent` 编码）。
- **UploadArea**：拖拽 + 点击选择；上传中遮罩 + Spin + 进度条。
- **ConfirmModal**：删除前确认。

### 7.3 状态管理

- `authStore`：token、用户名、`login/logout/restore`。
- `fsStore`：当前路径、列表数据、loading 状态、`uploading` 标志。

### 7.4 异步与 UX

- 所有操作均为 `async/await`；axios 拦截器：
  - 请求注入 token；
  - 401 → 清 token、跳登录页。
- 进入文件夹、刷新列表、删除、上传后均 await 完成再解锁 UI。

## 8. 测试方案

### 8.1 后端测试

| 类型 | 范围 | 工具 |
| --- | --- | --- |
| 单元测试 | `safePath`、`auth.service`、`fs.service` 纯函数 | Vitest |
| 集成测试 | 完整接口（登录、列表、新建、删除、批量删除、上传、静态访问） | Vitest + Supertest |

关键用例：
- 登录：正确账号密码 → 200 + token；错误密码 → 401；连续 5 次失败 → 429。
- 鉴权：无 token / 错误 token → 401；过期 token → 401。
- 列表：正常返回；路径穿越 `../../etc` → 400。
- 新建文件夹：正常；同名 → 400。
- 删除文件夹：空 → 200；非空 → 400。
- 删除文件 / 批量删除：成功；不存在 → 404。
- 上传：多文件成功；超大文件 → 413；目标目录不存在 → 400；同名追加后缀。
- 公开访问（根路径托管）：
  - `GET /a.txt` 返回文件内容；`GET /sub/b.png` 返回文件内容。
  - `GET /sub/`（目录）→ 404，且响应体不含目录列表。
  - `GET /../app.js`（穿越）→ 400。
  - `GET /.env`（dotfile）→ 404。
  - `GET /api/auth/login` 命中 API 路由（不会被静态兜底拦截）。
  - `GET /admin` 命中 SPA（不会被静态兜底拦截）。
  - 即使 `STATIC_ROOT/api/x.png` 实际存在，`GET /api/x.png` 也不会下发它（被 API 路由遮蔽，预期行为）。

### 8.2 前端测试

| 类型 | 范围 | 工具 |
| --- | --- | --- |
| 组件测试 | Login、FileTable、UploadArea、Breadcrumb | RTL + Vitest |
| 接口 mock | 所有 API | MSW |
| 流程测试 | 登录→进入文件夹→上传→批量删除 | RTL 模拟用户操作 |

关键用例：
- 登录页：表单校验、错误提示、成功后跳转。
- 列表：渲染正确；进入文件夹更新面包屑。
- 上传：上传中区域禁用、其他按钮禁用、完成后恢复并刷新列表。
- 删除：确认弹窗；空文件夹删除成功；非空文件夹返回错误时正确提示。

### 8.3 覆盖率目标

- 后端语句/分支覆盖率 ≥ 90%（路径安全相关 100%）。
- 前端语句覆盖率 ≥ 80%。
- CI 在 PR 上跑全量测试 + 覆盖率检查；阈值不达标即失败。

### 8.4 测试数据隔离

集成测试用临时目录（`os.tmpdir() + uuid`）作为 `STATIC_ROOT`，每个用例 `beforeEach` 创建、`afterEach` 清理，互不干扰。

## 10. 部署与运行

### 10.1 开发模式

```bash
pnpm install
pnpm gen:hash mypassword           # 生成密码哈希
cp .env.example .env               # 填入哈希、JWT_SECRET、STATIC_ROOT
pnpm -C server dev                 # 后端 nodemon 热重启，:3000
pnpm -C web dev                    # 前端 Vite dev server，:5173 代理 /api → :${PORT}（PORT 从根 .env 通过 loadEnv 读取，并经 define 暴露给前端为 import.meta.env.VITE_SERVER_PORT，用于"访问"链接拼 base URL）
```

### 10.2 生产模式

```bash
pnpm -C web build                  # 产出 web/dist
pnpm -C server start               # 后端托管 web/dist 与静态资源根目录
```

后端启动时按以下路由优先级装配（详见 §6.7）：

- `/api/*`           → 业务接口
- `/admin`、`/admin/*` → 前端 SPA（`web/dist`，history fallback 到 `index.html`）
- 其他 `GET/HEAD`    → 兜底，从 `STATIC_ROOT` 查找文件下发；目录或未命中 → 404

**说明**：若 `STATIC_ROOT` 下存在名为 `api` 或 `admin` 的顶级目录/文件，会被路由遮蔽而无法通过 URL 访问，由用户自行规避，后端不做主动校验。

## 11. 里程碑拆分
